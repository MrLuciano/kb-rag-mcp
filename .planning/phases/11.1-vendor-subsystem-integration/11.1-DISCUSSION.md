# Phase 11.1: Vendor/Subsystem Integration Completion

**Type:** Remediation (critical gap from milestone v1.2 audit)
**Status:** Discussion
**Priority:** Critical
**Estimated Effort:** 2-3 hours
**Dependencies:** Phase 11 (Auto-Classification)

---

## Problem Statement

Phase 11 successfully implemented vendor/subsystem classification and storage, but **failed to complete the integration** — these fields are invisible to end users:

**What Works:**
- ✅ `infer_vendor()` and `infer_subsystem()` classify documents correctly
- ✅ `ingest.py` extracts vendor/subsystem from `classify()` results
- ✅ Vendor/subsystem stored in Qdrant chunk payload during ingestion

**What's Broken:**
- ❌ `vector_store.search()` does not extract vendor/subsystem from Qdrant results
- ❌ MCP clients receive search results WITHOUT vendor/subsystem metadata
- ❌ No filter capability — users cannot search by vendor or subsystem
- ❌ MCP tool schema does not expose vendor/subsystem parameters

**Impact:** CLASSIFY-01 value proposition broken — classification data exists in database but is inaccessible to users.

---

## Discovered By

Milestone v1.2 audit (2026-05-27) via `/gsd-audit-milestone` workflow:
- Integration checker traced ingest → storage → search → results
- Found vendor/subsystem stored at `ingest.py:459-460` but missing from `vector_store.py:207-218` result extraction
- Verified MCP tool schema in `server.py:84-295` lacks vendor/subsystem parameters

---

## Root Cause Analysis

**Why did this happen?**

1. **Phase 11 scope creep:** Phase focused on classification logic (inference + enrichment) but did not include full E2E integration testing
2. **Missing VERIFICATION.md:** Phase 11 executed without formal verification artifact — no checklist to verify search results contain new fields
3. **Incomplete plan:** Phase 11 plans (11-01, 11-02) covered ingest pipeline changes but did not include vector store and MCP server changes

**Why wasn't it caught earlier?**

1. Phase 11 SUMMARY.md marked CLASSIFY-01 as complete based on "fields stored in Qdrant" — technically correct but insufficient acceptance criteria
2. No E2E flow test: "ingest document → search → verify result payload contains vendor/subsystem"
3. REQUIREMENTS.md traceability marked CLASSIFY-01 complete without verification evidence

## Scope

**In Scope:**
1. Add vendor/subsystem to search result extraction in `vector_store.py`
2. Add vendor/subsystem filter parameters to `VectorStore.search()` signature
3. Implement Qdrant `FieldCondition` filtering for vendor/subsystem (like product/doc_type)
4. Extend `search_kb` MCP tool schema with vendor/subsystem parameters
5. Add vendor/subsystem to `list_documents()` result extraction
6. Update integration tests to verify vendor/subsystem in results

**Out of Scope:**
- Hybrid search vendor/subsystem filtering (defer to Phase 11.2 if needed)
- Reranker awareness of vendor/subsystem (not needed for this phase)
- CLI `kb-ingest search` vendor/subsystem filters (no CLI search command exists)
- Documentation updates (will be covered in verification)

---

## Success Criteria

**Definition of Done:**

1. **Search results include vendor/subsystem:**
   - `vector_store.search()` extracts `vendor` and `subsystem` from Qdrant point payload
   - MCP clients receive these fields in result metadata

2. **Filtering works:**
   - `vector_store.search(vendor="OpenText")` returns only OpenText documents
   - `vector_store.search(subsystem="Archive")` returns only Archive subsystem documents
   - Combined filters work: `vendor="OpenText" AND product="WebReports"`

3. **MCP tool schema updated:**
   - `search_kb` tool inputSchema includes optional `vendor` and `subsystem` string parameters
   - `list_documents` tool inputSchema includes optional `vendor` and `subsystem` string parameters

4. **Backward compatibility:**
   - Existing search calls without vendor/subsystem parameters work unchanged
   - Documents without vendor/subsystem (legacy or external docs) return empty strings for these fields

5. **Test coverage:**
   - Unit tests verify vendor/subsystem extraction from Qdrant results
   - Integration tests verify vendor/subsystem filtering
   - E2E test: ingest → classify → search with filter → verify result payload

6. **Documentation:**
   - README.md search examples include vendor/subsystem filtering
   - MCP tool descriptions mention vendor/subsystem parameters

---

## Constraints

- **No breaking changes:** Existing search API calls must work unchanged
- **No schema migration:** Vendor/subsystem already exist in Qdrant payload (since Phase 11)
- **No ingest changes:** Classification logic is correct; only search path needs fixes
- **Maintain performance:** Adding 2 filter conditions should not degrade search latency
- **Follow Phase 11 patterns:** Use same FieldCondition logic as product/doc_type filtering

---

## Risk Assessment

**Low Risk:**
- No database schema changes
- No classification logic changes
- Additive changes only (new parameters, new extractions)
- Existing tests should pass unchanged

**Medium Risk:**
- If vendor/subsystem fields are missing from old documents → empty strings (acceptable)
- If Qdrant payload structure differs from expected → extraction fails silently (needs validation)

**Mitigation:**
- Test with real Qdrant data from Phase 11 ingestion
- Verify payload structure matches `ingest.py:459-460` expectations
- Add defensive code: `payload.get("vendor", "")` instead of `payload["vendor"]`
