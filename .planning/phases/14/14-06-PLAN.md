**Why:** Users need to specify custom storage locations for Qdrant data (external drives, NFS mounts, etc.).

<technical_notes>
**Why wget instead of curl:**
- qdrant/qdrant:latest is Alpine-based (busybox utilities)
- wget is included in Alpine base, curl is not
- `wget --spider -q` = silent HEAD request, perfect for healthchecks
- Exit code 0 on success, non-zero on failure (same as curl -f)

**Volume path substitution:**
- `${VAR:-default}` syntax is Docker Compose native (no special processing)
- Works with docker-compose v1.25.0+ (current requirement: v2.x)
- Supports relative paths (./data/qdrant), absolute paths (/mnt/storage), and Windows paths (C:\kb-data\qdrant via WSL2 path translation)

**Healthcheck timing:**
- interval: 30s = check every 30 seconds
- timeout: 10s = wait 10s for response before failing
- retries: 3 = 3 failures before marking unhealthy
- First check at startup, then every 30s
- Container marked healthy after first successful check
</technical_notes>
