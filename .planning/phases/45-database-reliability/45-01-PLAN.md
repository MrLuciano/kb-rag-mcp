<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite connection scope | Each `sqlite3.connect()` call creates a process-local file handle; no network boundary crosses here |
| Schema migration path | DDL executes against local on-disk SQLite files — no external input in the DDL strings themselves |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-01 | Tampering | Bare DDL in migrations | mitigate | `CREATE TABLE IF NOT EXISTS` makes all migration DDL idempotent — re-running a completed migration is a no-op instead of failing with "table already exists" (Task 4) |
| T-45-02 | Information Disclosure | Unclosed connections | mitigate | `with sqlite3.connect() as conn:` context managers guarantee timely connection closure, preventing file-handle exhaustion and stale read-locks on the WAL (Task 1) |
| T-45-03 | Elevation of Privilege | Unenforced foreign keys | mitigate | `PRAGMA foreign_keys=ON` on every connection enforces referential integrity — orphaned rows from the application layer get rejected (Task 2) |
| T-45-04 | Denial of Service | Missing indexes | mitigate | Indexes on `api_keys(prefix)` and `query_log(timestamp)` prevent full-table scans on high-traffic auth and analytics paths (Task 3) |
| T-45-SC | Tampering | Package installs | accept | No new package dependencies introduced in this phase — all changes use stdlib `sqlite3` |
</threat_model>
