    **Part A — Add `get_latency_stats()` to QueryAnalyzer**

    In `kb_server/analytics/query_analyzer.py`, add a new method `get_latency_stats(time_range_days=7) -> List[Dict[str, Any]]` following the existing pattern of `get_zero_result_queries()` and `get_most_common_queries()`.

    Method contract:
    - Accept `time_range_days: int = 7` parameter
    - Compute cutoff timestamp as `datetime.utcnow() - timedelta(days=time_range_days)` in ISO format
    - Execute SQL: `SELECT latency_ms FROM query_log WHERE timestamp >= ?` using the cutoff
    - Fetch all rows, extract non-None latency values, sort them
    - Compute p50, p95, p99 using linear interpolation between nearest ranks (standard percentile calculation)
    - Return a list containing a single dict: `{'operation': 'All queries', 'count': N, 'p50_ms': float, 'p95_ms': float, 'p99_ms': float}` (round each to 1 decimal place)
    - If no rows returned, return `[{'operation': 'All queries', 'count': 0, 'p50_ms': 0.0, 'p95_ms': 0.0, 'p99_ms': 0.0}]`
    - Follow existing patterns: manual `conn = sqlite3.connect()`, `cursor = conn.cursor()`, `rows = cursor.fetchall()`, `conn.close()` (no context manager, matching existing code style per D-05)
    - Log with `log.debug(...)` before returning
    - Import `datetime` inside the method (following existing pattern of no top-level datetime import; the test file is the reference boundary)

    Percentile calculation helper (inline, not a separate def):
    ```
    sorted_vals = sorted([row[0] for row in rows if row[0] is not None])
    n = len(sorted_vals)
    if n == 0: return default zero row
    idx = (n - 1) * p / 100
    lower = sorted_vals[int(idx)]
    upper = sorted_vals[min(int(idx) + 1, n - 1)]
    frac = idx - int(idx)
    return lower + (upper - lower) * frac
    ```

    Do NOT add `datetime` or `timedelta` at the top of the file — import inline in the method body to match existing codebase convention (the existing methods use no module-level imports beyond sqlite3, Path, List, Dict, Any).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin browser → Analytics route | Authenticated admin session loads query log data into HTML tables |
| QueryAnalyzer → SQLite query_log | Internal read-only SQL query — no user-controlled input in SQL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-42-01 | Information Disclosure | `/admin/tabs/analytics` route | accept | Analytics tab is behind admin auth (Phase 28b); data is aggregate query statistics, not PII. Acceptable risk for internal tool. |
| T-42-02 | Tampering | QueryAnalyzer SQL queries | mitigate | All SQL queries use parameterized placeholders (`?`) exclusively — no string interpolation or f-string SQL construction. `get_latency_stats()` follows this pattern with the `time_range_days` parameter as a SQL placeholder. |
| T-42-03 | Denial of Service | Heavy query_log table | accept | Query log schema has no index on `timestamp`; full table scan on every analytics load. Phase 45 (DB-03) will add an index on `query_log.timestamp`. Acceptable at current scale (< 100K rows). |
| T-42-SC | Tampering | Python packages (stdlib only) | mitigate | No new external dependencies added. Only `sqlite3`, `datetime`, `logging` (all stdlib) are used. No slopcheck needed. |
</threat_model>
