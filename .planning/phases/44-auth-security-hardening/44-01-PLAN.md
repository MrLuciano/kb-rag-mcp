<objective>
Fix six auth infrastructure gaps identified in the REVIEW.md audit: mount the auth router, add erasure separation of duties, enforce ownership checks, secure session cookies, batch verify_key writes, and hash rate-limit subjects.

**Purpose:** Eliminate security vulnerabilities in the auth subsystem — prevent privilege escalation (non-admin listing any user's keys or exporting any user's data), prevent session cookie leakage over HTTP, eliminate unnecessary DB writes on every API key verification (DOS vector), and avoid exposing API key prefixes in rate-limit tracking.

**Output:** Modified `server.py`, `router.py`, `service.py`, and `test_auth_api.py` with all 6 hardened behaviors verified by updated tests.
</objective>

    Every API key verification currently writes `last_used_at` and commits to the database. For a high-traffic server where every MCP tool call verifies the API key, this turns every read operation into a write, wasting I/O and risking contention. Per D-06, remove writes from `verify_key()` and add a separate `record_key_usage()` method called only from the session endpoint.

    **Changes in `kb_server/auth/service.py`:**

    1. **Remove `last_used_at` write from `verify_key()`** (lines 162-163). Delete these two lines:
       ```python
       api_key.last_used_at = datetime.utcnow()
       self._session.commit()
       ```
       The method now returns the user without any side effect. The `api_key` local variable (line 143) is still needed for the revoked check on line 147; ensure the variable is not removed.

    2. **Add `record_key_usage` method** to `AuthService` (after `verify_key`, before the audit logging section at line 166):
       ```python
       def record_key_usage(self, key_id: str) -> None:
           """Update last_used_at for a known API key ID.
           
           Called from the session endpoint after successful authentication,
           not from verify_key (which is called on every request).
           """
           api_key = (
               self._session.query(ApiKey)
               .filter(ApiKey.id == key_id)
               .first()
           )
           if api_key is not None:
               api_key.last_used_at = datetime.utcnow()
               self._session.commit()
       ```

    **Changes in `kb_server/auth/router.py`:**

    3. **Update `create_session`** (lines 53-82): After the user is authenticated (via `get_current_user` dependency), call `record_key_usage`. The session endpoint needs the raw API key to look up its ID. Read it from headers:
       
       At the top of the `create_session` function body, after getting `expires_at` and before constructing the session token, add:
       ```python
       # SEC-05: Record key usage (not on every verify_key call)
       raw_key = request.headers.get("X-API-Key", "")
       if not raw_key:
           auth_header = request.headers.get("Authorization", "")
           if auth_header.startswith("Bearer "):
       ```

       a. Add a `record_key_usage_by_raw_key(raw_key)` that hashes and looks up

       Then in `router.py`, the session endpoint directly calls:
       ```python

       ```

This leaks part of the raw API key into the rate-limiter's tracking data, exposing key prefixes in logs and metrics. Per D-07, hash the subject using `hashlib.sha256(subject.encode()).hexdigest()[:16]` before passing to `rate_limiter.check()`.

    **Changes in `kb_server/server.py`:**

    2. **Add a helper function** after the config constants (after line 78, before `_current_subject`):
       ```python
       def _hash_subject(subject: str) -> str:
           """SHA-256 hash a rate-limit subject for non-reversible tracking.
           
           Returns first 16 hex chars of the digest — sufficient uniqueness
           for rate-limit buckets without exposing raw API key prefixes.
           """
           return hashlib.sha256(subject.encode()).hexdigest()[:16]
       ```

    3. **In `call_tool`** (line 574): Hash the subject before passing to `rate_limiter.check()`:
       ```python
       subject = _hash_subject(_current_subject.get() or "unknown")
       ```

       Also hash the subject when setting the context var (line 1516):
       ```python
       _current_subject.set(_hash_subject(sse_subject))
       ```

    5. **In `handle_mcp` (streamable HTTP)** (line 1620): Hash `subject` before passing to `rate_limiter.check()`:
       ```python
       allowed, retry_after = await rate_limiter.check(_hash_subject(subject))
       ```
       Also hash when setting the context var (line 1607):
       ```python
       _current_subject.set(_hash_subject(subject))
       ```

    **This ensures** that `_current_subject` always holds the hashed value, so the tool-level check in `call_tool` (which reads from `_current_subject`) also uses the hashed subject. The rate-limiter subject gauge and prometheus metrics will see hashed identifiers instead of raw key prefixes.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Auth Router | Unauthenticated HTTP requests arrive at auth endpoints. The auth router's first gate is API key validation via `get_current_user`. |
| User → Another User's Data | Authenticated user must be prevented from accessing another user's API keys or personal data without admin role. |
| Erasure Approver → Executor | Admin who approves an erasure must not automatically execute it — separate endpoint enables separation of duties. |
| API Key → Rate-Limiter | Key prefixes must not leak into rate-limiter tracking data. The rate-limiter subject is hashed. |
