# Plan 51-01 — Document Tag Management & Re-ingest Control

**Phase:** 51 — Document Tag Management & Re-ingest Control  
**Objective:** Implement bulk tag editing via CLI and Web UI for correcting misclassified documents after ingestion.  
**Estimated Duration:** 6-8 hours  
**Tasks:** 6  
**Depends On:** Phase 28c (Admin SPA), Phase 45 (Registry bulk ops), Phase 16 (Reclassification)  
**Decisions Locked:** 13 (see CONTEXT.md D-01 through D-13)

---

## Locked Decisions Summary

| ID | Decision | Impact on Plan |
|----|----------|----------------|
| D-01 | Document-level tags | All chunks of a document share tags |
| D-02 | Visually separate from classification | Tags column distinct from product/type/version in UI |
| D-03 | Minimal validation: 50 chars, no whitespace, case-insensitive | Validation logic in API and CLI |
| D-04 | Both inline + dedicated /admin/tags UI | Add inline editing to existing documents table |
| D-05 | Queue re-ingest as background job | Use existing job infrastructure, not watcher |
| D-06 | Opt-in searchable (stored, not indexed) | Tags in Qdrant payload, search integration deferred |
| D-07 | Minimal audit log | user_id, timestamp, doc_path, action, values |
| D-08 | Lazy migration — no bulk script | Tags field added on first edit, no migration task |
| D-09 | Best-effort bulk operations | Apply successes, report failures in summary |
| D-10 | Plain Bootstrap badges | No color management in UI |
| D-11 | Max 20 tags per document | Enforced at API level |
| D-12 | Tag autocomplete | Dropdown with existing tags as you type |
| D-13 | Cascade deletion | Deleting a tag removes it from all documents |

## Task 2: CLI — `kb-rag tags` Command Group

**Files:** `ingest/cli/tags.py`, `ingest/cli/main.py`

**Steps:**
1. Create `ingest/cli/tags.py` with Click command group `tags_group`
2. Implement `kb-rag tags list`:
   - Query SQLite registry for distinct tag values + counts
   - Rich table output: Tag, Count, Products, Types
   - Support `--product`, `--type`, `--vendor` filters
3. Implement `kb-rag tags update`:
   - Arguments: `--add TAGS`, `--remove TAGS`, `--replace TAGS`, `--filter FIELD=VALUE`
   - `--dry-run` shows affected document count and sample IDs (per D-03, validate tags: 50 chars max, no whitespace, lowercase)
   - `--yes` bypasses confirmation
   - Best-effort: apply successes, report failures (per D-09)
   - Calls `VectorStore.update_tags()` under the hood
4. Implement `kb-rag tags remove`:
   - Arguments: `--filter FIELD=VALUE` (e.g., `product=BadProduct`)
   - `--dry-run` preview
   - Deletes from Qdrant + registry
   - Requires `--yes` or interactive confirmation
5. Implement `kb-rag tags reingest`:
   - Arguments: `--filter FIELD=VALUE`
   - Creates background job (per D-05, NOT just setting status=pending)
   - Deletes Qdrant chunks for matched files
   - Job appears in existing Jobs tab
6. Implement `kb-rag tags delete-tag` (cascade per D-13):
   - Arguments: `--tag TAG_NAME`
   - Removes tag from ALL documents that have it
   - `--dry-run` shows affected count
   - Requires `--yes`
7. Register `tags_group` in `ingest/cli/main.py`

**Must-haves:**
- [ ] `kb-rag tags list` shows tag counts with filters
- [ ] `kb-rag tags update --add/remove/replace` works with --dry-run
- [ ] `kb-rag tags remove` deletes from both Qdrant and registry
- [ ] `kb-rag tags reingest` queues background job + deletes chunks
- [ ] `kb-rag tags delete-tag` cascades to all documents (D-13)
- [ ] Tag validation: 50 chars max, no whitespace, lowercase (D-03)
- [ ] Best-effort error handling (D-09)
- [ ] Max 20 tags per document enforced (D-11)

**Tests:**
- `test_cli_tags.py` — 10 tests: list, update dry-run, update apply, remove dry-run, remove apply, reingest, delete-tag, filter parsing, confirmation prompt, validation errors

## Task 4: Validation, Audit & Error Handling

**Files:** `ingest/cli/tags.py`, `kb_server/ui/routes_admin.py`, `ingest/core/metadata.py`

**Steps:**
1. **Tag validation (D-03, D-11):**
   - Max 50 characters
   - No whitespace allowed
   - Case-insensitive (normalize to lowercase)
   - Max 20 tags per document
   - Prevent duplicate tags on same document
2. **Audit logging (D-07):**
   - Log to `tags_history` table in registry: user_id, timestamp, document_path, action (add/remove/replace/delete-tag), tag_values
   - One entry per document mutation
3. **Best-effort error handling (D-09):**
   - Apply what succeeds, report failures at end
   - Summary format: "97 updated, 3 failed: [list]"
   - Handle Qdrant filter syntax errors with user-friendly messages
   - Handle registry/Qdrant inconsistency gracefully
4. **Empty state handling:**
   - Show "No tags yet" message when document has no tags
   - Show "No documents match this filter" when filter returns empty

**Must-haves:**
- [ ] Tag validation enforced at API and CLI (D-03, D-11)
- [ ] Audit logging for all mutations (D-07)
- [ ] Best-effort bulk operations with failure summary (D-09)
- [ ] Graceful handling of empty state
- [ ] User-friendly error messages

**Tests:**
- `test_tags_integration.py` — 4 tests: validation, max 20 limit, audit log, best-effort errors
